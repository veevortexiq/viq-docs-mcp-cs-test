    Frequently Asked Questions

    Common questions about BigCommerce product migration.

    General Questions

    What happens to my original store data?

    Your source store data is never modified or deleted. The migration only reads data from your source store and creates copies in the destination store. Your source products remain exactly as they are.

    How long does a migration take?

    Migration time depends on your product complexity and catalog size:

    Product Type

    Approximate Time

    Simple product (no variants)

    5-15 seconds

    Product with variants

    15-30 seconds

    Complex product (modifiers, custom fields, reviews)

    30-60 seconds

    Product with locale data

    1-2 minutes

    For bulk migrations:

    Catalog Size

    Approximate Time

    10 products

    5-15 minutes

    100 products

    30-60 minutes

    500 products

    2-4 hours

    1,000+ products

    Several hours

    Note: Products with many images or variants take longer.

    Can I migrate multiple times?

    Yes! The platform uses duplicate detectionif a product already exists in the destination (matched by name or SKU), it's updated rather than duplicated. This means you can safely run migrations multiple times without creating duplicate products.

    What if my migration fails partway through?

    • Already-migrated products remain in the destination store

    • The failed product can be retried individually

    • Detailed error information helps identify the issue

    • If updating an existing product, you can trigger a rollback to restore the original data

    Can I rollback a migration?

    Yes, for products that were updated (not newly created): - A backup is created before any changes - You can restore the product to its pre-migration state - Rollback restores: product base, options, variants, modifiers, custom fields, bulk pricing, and videos - Images are NOT rolled back (they're only added, never deleted)

    Component Cleanup (UPDATE Mode)

    What happens to existing product data when updating?

    When a product already exists in the destination store (matched by name or SKU), the migration operates in UPDATE mode. This involves a complete cleanup of existing components before recreating them from the source.

    Why does the platform delete existing components?

    This ensures: - Clean slate: No orphaned or duplicate data - Accurate transfer: Source becomes single source of truth - Correct mappings: New IDs are properly linked (options to variants, modifiers to complex rules) - No conflicts: Prevents ID mismatches between old and new data

    What gets deleted in UPDATE mode?

    Component

    What Happens

    Images

    ALL existing images deleted, then source images uploaded

    Options

    ALL options deleted (variants auto-delete with them)

    Variants

    Deleted when options are removed, then recreated

    Modifiers

    ALL modifiers deleted, then recreated from source

    Custom Fields

    ALL custom fields deleted, then recreated

    Bulk Pricing Rules

    ALL rules deleted, then recreated

    Videos

    ALL videos deleted, then recreated

    What about complex rules during cleanup?

    Complex rules are handled intelligently: - Matching rules: If a rule in destination matches source, it's preserved - Auto-generated rules: Single-condition rules (auto-created by BigCommerce) are cleaned up - Non-matching rules: Created fresh from source

    Is my data backed up before deletion?

    Yes! Before any deletion occurs: 1. Complete backup is created 2. Includes: product base, options, variants, modifiers, custom fields, bulk pricing, videos 3. Backup retained for rollback capability

    Can I prevent the cleanup?

    No. The cleanup is essential for accurate migration. However: - Backup protects you: Rollback available if needed - CREATE mode: If product doesn't exist, no cleanup needed - Test first: Migrate a sample product to verify results

    What if I only want to update some fields?

    Currently, migration is all-or-nothing for components. If you need selective updates: 1. Export the product data you want to preserve 2. Run the migration 3. Manually restore specific fields if needed

    Tip: For partial updates, consider using BigCommerce's native import/export tools instead.

    Products & Catalog

    Will my product images transfer?

    Yes. Product images are downloaded from your source store and re-uploaded to the destination store. This ensures: - Images are properly hosted in your new store - All image URLs work correctly - Alt text and display order are preserved

    How is inventory handled?

    Inventory levels are transferred directly: - Product-level inventory for simple products - Variant-level inventory for products with options - Inventory warning levels are preserved - Inventory tracking settings (none, product, variant) are maintained

    What about products in multiple categories?

    Product category assignments are fully preserved: - All category assignments transfer - Category hierarchy is recreated in the destination - If a category doesn't exist, it's created with the full parent path

    Example:

    Product in: Electronics > Computers > Laptops
All three categories created if needed, product assigned to "Laptops"

    Are product metafields migrated?

    Yes, both custom fields and metafields are migrated:

    Custom Fields (displayed on storefront): - Field names and values preserved - Display order maintained

    Metafields (key-value metadata): - Product metafields fully migrated - Variant metafields migrated to corresponding variants - Category metafields migrated when categories are created - Namespace and permission settings preserved

    Note: App-specific metafields may require the same app installed in destination to be fully functional.

    What happens to SEO data?

    All SEO data transfers directly to help preserve your search engine rankings: - Page title - Meta description - Search keywords - Custom URL path

    Variants & Options

    How are option values mapped?

    When options are created in the destination store, BigCommerce assigns new IDs. The migration platform:

    1. Creates options in the destination (e.g., "Size" with values Small, Medium, Large)

    2. Records the mapping: Source ID Destination ID

    3. Uses this mapping when creating variants

    This ensures variants are correctly linked to their options.

    What if a variant's SKU already exists?

    If you're creating a new product: - All variants are created with their original SKUs - If a SKU conflict exists with another product, the variant creation may fail

    If you're updating an existing product: - Old variants are deleted first - New variants are created with source SKUs

    Are variant images preserved?

    Yes. Each variant can have its own image: - Variant-specific images are migrated - Images are linked to the correct variant after migration - Main product images are also preserved

    What about option display types?

    Pricing & Inventory

    Is pricing data preserved exactly?

    Yes, all pricing fields transfer exactly as configured: - Base price - Sale price - Cost price (your cost, hidden from customers) - Retail price (MSRP) - Variant-specific pricing overrides

    How does bulk pricing work after migration?

    Bulk pricing rules transfer completely: - All quantity tiers preserved - Discount types (percent off, fixed amount, set price) maintained - Rules apply immediately in the destination store

    Example:

    Buy 10-24: 10% off
Buy 25-49: 15% off
Buy 50+: 20% off

    All three rules migrate and work immediately.

    What about inventory levels?

    Inventory is copied at the time of migration: - Current stock levels transfer - Inventory warning thresholds transfer - Tracking settings (none, product, variant) preserved

    Note: Inventory is a snapshotchanges in the source store after migration won't automatically sync.

    Do modifier price adjustments work?

    Yes. Price and weight adjustments on modifier values transfer exactly: - Checkbox that adds $10 Works in destination - Dropdown with different prices per option All prices preserved

    What about custom fields?

    All custom fields migrate: - Field names preserved - Field values preserved - Display order maintained

    Example:

    Material: 100% Cotton
Care: Machine wash cold
Made In: USA

    All three fields appear on the destination product.

    Complex Rules

    What are complex rules?

    Complex rules are conditional logic rules that trigger actions based on modifier selections. They can: - Adjust price when certain modifier combinations are selected - Adjust weight for shipping calculations - Disable purchasing for certain combinations - Display different images based on selections

    Are complex rules migrated?

    Yes! Complex rules are fully migrated with: - All conditions (modifier + value combinations) - Price adjusters (percent, fixed amount, or absolute) - Weight adjusters - Purchasing disabled flags and messages - Enabled/stop flags

    How are modifier IDs handled in complex rules?

    The platform automatically maps modifier IDs: 1. Modifiers are created first in destination 2. Source-to-destination ID mapping is captured 3. Complex rules are updated with new modifier IDs 4. Modifier value IDs are also mapped correctly

    What if a complex rule references a modifier that wasn't migrated?

    If a modifier ID in a rule can't be mapped: - A warning is logged - The rule may not function correctly - Check that all modifiers were successfully migrated

    Do complex rules work immediately after migration?

    Yes. Once migration completes: - Rules are active in destination - Price/weight adjustments apply on product pages - Purchasing restrictions are enforced

    What about auto-generated complex rules?

    BigCommerce auto-generates simple complex rules for modifiers with price adjustments. During migration: - Matching auto-generated rules: Preserved (not duplicated) - Redundant auto-generated rules: Cleaned up to avoid conflicts - Custom multi-condition rules: Created fresh from source

    Can I test complex rules after migration?

    Yes! To verify complex rules work correctly: 1. Go to product page in destination store 2. Select modifier combinations that should trigger rules 3. Verify price/weight changes in cart 4. Test purchasing restrictions if applicable

    Are all metafield types supported?

    What about metafield permissions?

    Are variant metafields migrated?

    Yes. Variant metafields are: 1. Fetched from source variants 2. Mapped to corresponding destination variants (via SKU/option matching) 3. Created on the new variants

    What about category metafields?

    When a category is created during migration, its metafields are also migrated: - Existing category metafields in destination are replaced - Source metafields are created on destination category

    Reviews

    Are product reviews migrated?

    Yes, customer reviews are fully migrated: - Review title and content - Star rating (1-5) - Reviewer name and email - Original review date - Review status (approved, pending, disapproved)

    Do review ratings calculate correctly?

    Yes. After migration: - All reviews are attached to the destination product - Average rating is calculated from migrated reviews - Review count reflects total migrated reviews

    Can I migrate reviews selectively?

    Reviews are migrated as part of the product. If you migrate a product, its reviews come along. You cannot currently migrate only reviews without the product.

    Localization (Multi-Language)

    Are multi-language products supported?

    Yes! The platform migrates locale-specific data across 10 categories:

    1. Basic Information - Translated name and description

    2. SEO - Locale-specific page title and meta description

    3. URL - Custom URL path per locale

    4. Options - Translated option names and values

    5. Modifiers - Translated modifier names and values

    6. Custom Fields - Translated field names and values

    7. Images - Locale-specific alt text and visibility

    8. Pre-Order - Translated pre-order messages

    9. Storefront - Warranty, availability, search keywords

    10. Channel Overrides - Featured status per channel

    What locale data gets migrated?

    For each locale configured in your source store: - All text content that has locale-specific overrides - Locale-specific URL paths for SEO - Option and modifier label translations - Image alt text translations

    Do I need to set up locales in the destination first?

    Locale configuration should be set up in your destination store's settings. The migration transfers the locale-specific content, but your store must be configured to display those locales.

    Brands & Categories

    How are brands handled?

    Brands are matched by name: 1. Source product has brand "AudioTech" 2. Platform searches destination for "AudioTech" 3. If found: Uses existing brand 4. If not found: Creates new brand

    What about category hierarchy?

    Categories are recreated with full hierarchy:

    Example:

    Source category path: Electronics > Audio > Headphones

Migration creates (if needed):
1. "Electronics" (top level)
2. "Audio" (child of Electronics)
3. "Headphones" (child of Audio)

Product assigned to "Headphones"

    Are category descriptions migrated?

    When a category is created, only the essential information is migrated: - Category name - Parent relationship

    Category descriptions, images, and SEO data are not part of product migration. For full category migration, a separate process would be needed.

    Widget Templates & Custom Templates

    What are widget templates?

    Widget templates control how product page widgets appear. If your source product uses custom widgets, the template definitions are migrated to ensure the destination product displays correctly.

    What about custom page templates?

    If your source product uses a custom page template (e.g., product-special.html), the association is recreated in the destination.

    Important: The template file itself must exist in your destination theme. Only the association is migrated, not the template file.

    Same-Store Channel Migration (Multi-Storefront)

    What happens when source and destination stores are the same?

    When you migrate between channels in the same store, the platform automatically uses a simplified channel migration workflow. This is detected when: - Source store ID = Destination store ID - Source channel ID Destination channel ID

    In this mode, only channel-specific content is migrated because the product already exists.

    Why aren't product components migrated in channel migration?

    In same-store channel migration: - The product already exists in the store - Options, variants, modifiers, etc. are store-level data, not channel-specific - Only channel-specific data needs to be set up for the new channel

    What IS migrated: - Category assignments (to destination channel's tree) - Channel assignment (makes product visible) - Widget templates - Custom template associations - Locale data for the destination channel

    What is NOT migrated: - Product base data (already exists) - Options and variants (store-level) - Modifiers and complex rules (store-level) - Images and videos (store-level) - Custom fields, bulk pricing, reviews (store-level)

    Can I rollback a channel migration?

    No, channel migration does not include backup/rollback because: - Nothing is deleted: Original product data unchanged - Non-destructive: Only adds visibility to new channel - Reversible manually: Remove channel assignment via BigCommerce admin

    To "undo" a channel migration: 1. Go to BigCommerce admin 2. Edit the product 3. Remove the channel assignment for the destination channel

    How fast is channel migration compared to full migration?

    Channel migration is significantly faster:

    Product Type

    Full Migration

    Channel Migration

    Simple product

    5-15 seconds

    2-5 seconds

    With locale data

    1-2 minutes

    15-30 seconds

    Complex product

    30-60 seconds

    5-10 seconds

    Will channel migration create a duplicate product?

    No. Channel migration uses the same product ID. The product is simply made visible on an additional channel. You end up with one product visible on multiple channels, not multiple products.

    Can I migrate to multiple channels at once?

    Currently, you migrate to one destination channel per request. To publish a product to multiple channels: 1. Run channel migration for Channel A 2. Run channel migration for Channel B 3. And so on...

    Each migration is fast, so publishing to multiple channels takes minimal time.

    What about channel-specific pricing?

    Channel migration does not change pricing. If you need different prices per channel: 1. First, run the channel migration to make the product visible 2. Then, use BigCommerce's channel-specific pricing features to set prices per channel

    Does locale data differ between channels?

    Yes! Each channel can have its own locale-specific content. During channel migration, locale data from the source channel is copied to the destination channel. After migration, you can customize locale content per channel.

    Troubleshooting

    "Product already exists" - is this an error?

    No! This is expected behavior. It means: - A product with the same name or SKU was found in the destination - The platform will UPDATE this product instead of creating a duplicate - This prevents duplicate products in your catalog

    Important: UPDATE mode will delete and replace existing components. See Component Cleanup for details.

    Some product images didn't migrate - why?

    Common reasons: - Source image unavailable: The image URL might be broken or inaccessible - Image too large: Very large images may timeout during download - Format issues: Some image formats may not be supported

    Solution: Check the source product images are accessible and retry the migration.

    My variants aren't showing the right options

    This usually indicates an option mapping issue. Possible causes: - Options were modified in the destination after migration - Source product had inconsistent option values

    Solution: Re-migrate the product to refresh the option mappings.

    Modifiers aren't showing price adjustments

    Verify that: 1. The source product had price adjustments configured 2. The modifier type supports price adjustments (some don't) 3. The adjustment values weren't zero

    Complex rules not working after migration

    Check the following: 1. Modifiers migrated: Complex rules depend on modifiers existing first 2. ID mapping: If modifier migration failed, rule conditions may reference invalid IDs 3. Rule enabled: Verify the rule's "enabled" flag is true 4. Conditions met: Test the exact modifier combination that should trigger the rule

    Solution: Re-migrate the product to ensure modifiers and rules are properly linked.

    Product not appearing on storefront

    Check these settings: - Is Visible: Product visibility flag - Channel Assignment: Product must be assigned to the storefront channel - Category: Product should be in a visible category - Inventory: If tracking is enabled, verify stock quantity

    Custom fields not displaying

    Custom fields may not display automatically. Verify: 1. Custom fields were migrated (check product in admin) 2. Your theme is configured to display custom fields 3. The field names match what your theme expects

    I lost data after updating a product - can I recover it?

    If a product was updated (not created), a backup was taken: 1. Use the rollback feature with the migration ID 2. Original product data will be restored 3. Components restored: base data, options, variants, modifiers, custom fields, bulk pricing, videos

    Note: Images and reviews are NOT rolled back (they're only added, never deleted).

    Why are my destination product variants all gone?

    In UPDATE mode, all options are deleted before migration, which automatically deletes all variants. This is expected behavior: - Variants are recreated from source - Inventory levels from source are applied - Option value mappings are fresh

    If source had fewer variants than destination, only source variants will exist after migration.

    Metafields from apps aren't working

    App-specific metafields may require: 1. Same app installed in destination store 2. App to have matching client ID 3. Proper permission settings

    If the creating app isn't installed, metafields exist but may not function as expected.

    Best Practices

    Before Migration

    1. Verify source data: Ensure products have complete data

    2. Check images: Confirm all image URLs are accessible

    3. Review inventory: Know your current stock levels

    4. Document custom templates: Note which templates your products use

    During Migration

    1. Start small: Test with a few products first

    2. Monitor progress: Check migration status regularly

    3. Review results: Verify migrated products look correct

    After Migration

    1. Spot check products: Review several products in detail

    2. Test purchasing: Add products to cart, check pricing

    3. Verify images: Confirm all images display correctly

    4. Check variants: Test option selection and variant switching

    5. Review SEO: Confirm page titles and meta descriptions
